Critical — fix immediately
Public GitHub repo exposes architecture critical
The entire codebase — Supabase table names, RLS policy logic, API route patterns, auth flows, and business logic — is publicly visible. An attacker can map your entire data model before making a single request. Make the repo private immediately, or add a .env.example and audit what's safe to expose.
TypeScript error logs committed to repo critical
ts_errors.log and tsc_errors.log are committed to the main branch. These files expose internal file paths, type errors, component names, and system architecture details. Add them to .gitignore and purge from git history with git filter-repo.
Debug scripts in production root critical
check-schema.js and proxy.ts sit in the project root and are visible in the public repo. Schema-checking scripts reveal your database structure. The proxy script's purpose is unclear — if it forwards requests anywhere, it's a potential SSRF vector. Remove from root and .gitignore.
MCP config file exposed critical
.mcp.json is committed. This exposes MCP server configurations which may include internal tool endpoints. Add to .gitignore immediately.
No confirmed payment security layer critical
Supabase is not PCI DSS compliant for storing card data — it's a database, not a payment processor. There is no confirmed Razorpay, Stripe, or CCAvenue integration in the public codebase. If customers are placing orders and entering payment details, the payment path must be verified immediately. Never store card numbers in Supabase rows.
High severity — fix this week
HTTP security headers not configured in next.config.mjs high
Unlike Shopify, Next.js does not add security headers by default. Headers like Content-Security-Policy, X-Frame-Options: DENY, X-Content-Type-Options: nosniff, Referrer-Policy, and Strict-Transport-Security must be explicitly added in the headers() function in next.config.mjs. Verify via securityheaders.com.
Supabase RLS policies may have gaps high
The Supabase schema is visible in the public repo. If any RLS policy is misconfigured (e.g., allowing anon reads on orders or user_addresses), attackers can dump customer data via the public Supabase anon key. Run Supabase's RLS advisor and test every table with an unauthenticated client.
Deployment confusion — both Netlify and Vercel configs present high
Both netlify.toml and vercel.json exist in the repo, along with a netlify/functions/ folder. If both are ever live simultaneously, environment variables and serverless functions may behave inconsistently. Confirm which is the canonical deployment and remove the other's config.
Email domain records unverified (SPF/DKIM/DMARC) high
If order confirmation emails are sent from @haxeus.in, missing SPF/DKIM/DMARC records mean any attacker can spoof your domain. This destroys customer trust and can get your domain blacklisted. Verify at mxtoolbox.com.
Security action plan (priority order)
1
Make the GitHub repo private right now
Go to Settings → Danger Zone → Change visibility → Private. Then rotate any credentials that were ever in the repo's history — Supabase service role key, any API keys, payment keys.
2
Purge log files from git history
Run git filter-repo --invert-paths --path ts_errors.log --path tsc_errors.log, then force push. Add both to .gitignore.
3
Add security headers to next.config.mjs
Add a headers() async function returning CSP, X-Frame-Options, X-Content-Type-Options, HSTS, and Referrer-Policy for all routes (source: '/(.*)'). Score on securityheaders.com — target A.
4
Audit every Supabase RLS policy with a fresh anon client
Open a new browser in incognito, hit your Supabase project's REST API directly with the anon key, and try to SELECT from orders, user_addresses, profiles. Any that return data = RLS breach.
5
Verify or integrate a payment gateway
Use Razorpay (best for India, supports UPI/Cards/Wallets) or Stripe. Never store card details in Supabase. Handle all card data server-side in a Supabase Edge Function that calls the payment gateway API, then store only the payment intent ID.
The core problem — Next.js 15 SEO requires explicit implementation
No generateMetadata exports detected
Next.js App Router does not inject meta titles or descriptions automatically. Every page.tsx needs either a static export const metadata or an async export async function generateMetadata(). Without this, Google sees blank or generic titles — a ranking killer.
No sitemap.xml route found
Next.js 15 supports automatic sitemaps via app/sitemap.ts (returns a MetadataRoute.Sitemap). There is no evidence this exists. Alternatively, install next-sitemap. Without a sitemap, Googlebot has no map of your product pages.
No robots.txt confirmed
Create app/robots.ts that exports a MetadataRoute.Robots object, or add a static public/robots.txt. Without it, some crawlers may default to not indexing.
Zero backlinks, zero brand mentions
HAXEUS does not appear in any Indian streetwear roundup, media outlet, influencer feed, or fashion directory. Domain authority is effectively zero. No organic traffic possible until this changes.
Dynamic product pages need structured data
Product pages (app/products/[id]/page.tsx) should include JSON-LD structured data (Schema.org Product type) with name, image, price, and availability. This enables Google Shopping-style rich results in SERPs.
Image alt text on Supabase Storage images
Product images served from Supabase Storage need descriptive alt text. The Next.js Image component's alt prop must come from your product data, not be hardcoded as empty or generic.
Next.js-specific SEO fixes (priority order)
1
Add metadata to every page.tsx
At minimum: homepage, products listing, each product detail page, about, cart, and auth pages. For product pages use generateMetadata that fetches the product name and description from Supabase and returns it as the title/description. Target format: "HAXEUS | [Product Name] — Premium Artistic Streetwear India".
2
Create app/sitemap.ts and app/robots.ts
The sitemap should dynamically fetch all product IDs from Supabase and generate URLs for each. Set a weekly revalidate interval. Submit the sitemap URL to Google Search Console immediately after deploying.

In each product page.tsx, add a <script type="application/ld+json"> with Schema.org Product markup — name, image, price, currency (INR), availability, and brand (HAXEUS). This unlocks rich results and Google Shopping visibility.
4
Set up Google Search Console today
Verify ownership via HTML tag in app/layout.tsx (use Next.js metadata verification field). Submit sitemap. Monitor indexing errors daily for the first two weeks.
5
Start a blog using Next.js MDX or a CMS
Add a app/blog/ route using MDX files or connect Sanity/Contentlayer. Target keywords: "artistic streetwear India", "graphic tees India", "premium streetwear [city]". Two posts per month minimum. This is your primary organic traffic engine.
UX issues — confirmed from codebase
_disabled/ folder signals incomplete features
A _disabled/ directory in the root suggests features were built then stripped — possibly loyalty program, reviews, or coupons. Incomplete features create broken user journeys if any UI elements still reference them.
No review system visible in live flow
Reviews are listed as a feature in the README but require custom Supabase tables + UI. There is no third-party review widget integration (unlike Shopify where Judge.me installs in 2 clicks). This must be built or integrated via a headless reviews API.
Cart context — server/client hydration risk
The CartContext uses React Context (client-side only). In Next.js 15 App Router, server components can't access context. Cart state must be in a Client Component boundary. If cart total or item count shows 0 on first render, this is the cause — add skeleton loaders.
No size guide visible
Indian buyers are extremely size-sensitive when buying online. Absence of a size chart is one of the top reasons for cart abandonment on clothing stores. This must be on every product page.
TypeScript errors in production codebase
Both ts_errors.log and tsc_errors.log being non-empty at commit time suggests the build has TypeScript issues being suppressed. These can become runtime errors. Run npx tsc --noEmit and fix all errors before next deployment.
Conversion improvements
No trust signals on product pages
Indian buyers need to see: (a) real customer reviews/photos, (b) a clear return/exchange policy with timeline, (c) COD availability badge, (d) estimated delivery date. Without these, conversion rates for a new brand will be under 0.5%.
No email capture / welcome flow
There is no confirmed pop-up or inline form to capture emails on first visit. Every visitor who leaves without buying is lost forever without a follow-up mechanism. Add an exit-intent or timed pop-up with a 10% discount offer.
No WhatsApp integration
WhatsApp Business broadcasts have 60–80% open rates in India. Add a WhatsApp chat button (floating widget) and build an opt-in flow for order updates and drop announcements. This is the single highest-ROI retention channel for Indian D2C brands.
Framer Motion animations may hurt mobile performance
Heavy Framer Motion animations on product pages can drop Lighthouse Performance scores on mobile below 50. Profile on a mid-range Android device. Use prefers-reduced-motion media query and lazy-load non-critical animations.
Drop model / countdown timer not implemented
The codebase has no visible countdown timer component for limited drops. This is the highest-urgency UX addition — drop launches with scarcity timers drive 3–5× higher conversion in Indian streetwear.

India's apparel market heads toward $124B by 2029. Streetwear is the fastest growing segment, led by Gen Z buyers who strongly prefer domestic brands with cultural authenticity.
Drop model outperforms continuous inventory
Urban Monkey, Fugazee, and Jaywalking all use limited drops. Scarcity drives repeat visits, social sharing, and FOMO-based conversion. This model also reduces inventory risk massively.
